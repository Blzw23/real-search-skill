# 深度调研流程

## 0. 理解意图

把用户的话改写成可执行研究目标：

- 用户为什么要研究这个？
- 是为了学习、选型、创业/项目构思、写文章、做产品，还是实现代码？
- 用户已有知识边界是什么？
- 用户能接受多长、多深、多自主？

如果用户说不清楚，先给默认研究框架，不要卡住。

## 1. 建立领域地图

第一轮目标是避免视野过窄：

- 搜索官方文档、GitHub、论文、survey、awesome list、benchmark、成熟产品。
- 找出 10-20 个候选对象。
- 记录每个对象的一句话定位和为什么相关。
- 初步分层：成熟工程框架、研究原型、商业产品、协议/标准、论文方法。

自动发现脚本可以补充 GitHub、arXiv、OpenAlex、Crossref、Semantic Scholar 和若干搜索入口。注意：

- 自动发现结果只是候选资料，不是正式证据。
- 商业产品、官网搜索、benchmark 搜索、Awesome list 和社区讨论默认标记为 `D-待验证` 或 `C-社区/博客`。
- 真正进入报告的正式判断，必须回到官方文档、论文正文、源码路径或可复核 benchmark。

## 2. 选择深读对象

优先级：

1. 用户点名的对象。
2. Star/使用量/社区影响力高的开源项目。
3. 官方或论文影响力强的方案。
4. 与用户落地方向最接近的轻量项目。
5. 能形成差异化判断的反例。

## 3. 读文档

读文档时先找：

- README / overview
- architecture / concepts
- quickstart / examples
- API reference
- deployment / security / permission
- tracing / logging / eval
- changelog / roadmap

每读完一个对象，立刻写中文笔记，避免堆到最后。

## 4. 读源码

大仓库使用：

```bash
git clone --depth=1 --filter=blob:none <repo-url> 外部源码/<name>
```

如果仓库巨大，优先 sparse checkout 文档、examples、runtime、核心模块。

读源码时记录：

- clone URL
- commit hash
- 阅读的文件路径
- 核心调用链
- 发现的抽象
- 没读到的部分

## 5. 读论文

论文优先级：

- foundational paper
- recent survey
- benchmark/evaluation paper
- 与工程对象直接相关的 technical report
- 安全、可靠性、记忆、多智能体等专题论文

论文不要孤立总结，要和工程方案互相映射。

PDF 处理原则：

- 能下载 PDF 时先做基础检查：文件是否存在、大小是否异常、是否有 PDF header、能否读取页数。
- 能用 `pdftotext` 时抽取正文并记录字符数、章节数和抽取质量。
- 如果正文很短、章节识别很少或疑似扫描版，标注 `抽取质量低/需复核` 或 `疑似扫描版/待 OCR`。
- 不要因为有 PDF 文件或有少量抽取文本，就声称已经完整读懂论文。
- 本 skill 不默认内置 OCR；遇到扫描版时写入 `待人工阅读/待 OCR`。

## 6. 对比和收敛

对比维度可以按任务调整，常见维度：

- 定位
- 目标用户
- 语言/技术栈
- 核心抽象
- 扩展机制
- 工具系统
- 状态/记忆
- 多 Agent
- 权限/沙箱
- trace/log/eval
- 错误恢复
- 学习成本
- 可借鉴点
- 不适合点

最后必须回答：

- 现有方案是否已经满足用户目标？
- 如果不满足，缺口在哪里？
- 用户应该直接用、改造、组合，还是新做？
- 下一步最小可行动作是什么？

## 7. 完成前复盘

正式结束前做一次“从资料到判断”的复盘：

- 哪些判断证据最强？
- 哪些结论只是推测？
- 哪些资料还没读但值得后续读？
- 是否产出了下一步可执行方案？

重要调研建议额外生成内容复核清单：

```bash
python3 <skill_dir>/scripts/check_quality.py <workspace> --write-report --write-review
```

如果这份调研要用于选型、汇报或项目立项，再由 reviewer 角色逐条确认结论是否真的有证据。

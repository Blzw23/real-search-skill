# 现代 Agent 基座调研报告

## 摘要

成熟 Agent 生态已经有大量强框架，但它们大多落在四类：产品型本地助理、图运行时、高层业务 DSL、专用 coding agent。对算法工程师来说，仍存在一个机会：用 Python 做一个轻量、透明、可替换、能学习现代 Agent 方法的 runtime kernel。

## 关键结论

1. Agent 不应被理解为 prompt wrapper，而是 runtime。
2. Tool 层应设计成 Agent-Computer Interface，输出要短、结构化、可 trace。
3. Skill 与 MCP 是互补关系：MCP 是工具协议，Skill 是使用方法和经验。
4. 多 Agent 不是聊天室，MVP 应优先支持 manager-as-tool、reviewer 和 artifact-based collaboration。
5. Trace/eval/permission 应该从第一版进入设计。

## 主流成熟方案

| 类型 | 代表 | 值得学习 |
| --- | --- | --- |
| 产品型本地助理 | OpenClaw、Claude Code、Goose | skills、MCP、permissions、local-first |
| SDK 型框架 | OpenAI Agents SDK、PydanticAI | Agent/Runner/Tool API、structured output |
| 图运行时 | LangGraph | checkpoint、durable execution、state graph |
| 软件工程 Agent | OpenHands、SWE-agent、Aider | sandbox、trajectory、ACI、repo map |
| 多 Agent DSL | AutoGen、CrewAI | role、handoff、agent-as-tool、task orchestration |

- `PermissionPolicy`

## 风险与不确定项

- 如果过早做成完整平台，会复制 OpenHands/OpenClaw 的复杂度。
- 如果直接基于 LangGraph，学习曲线和抽象负担可能偏重。
- Goose、PydanticAI、CrewAI 仍需第二轮源码深读验证。

## 建议下一步

从 Python MVP 开始，实现单 Agent loop、ToolRegistry、SkillRegistry、MemoryStore、TraceSink、PermissionPolicy，再增加轻量 planner/worker/reviewer 编排。

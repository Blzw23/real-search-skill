# 质量门禁

完成前先验证，再声明完成。

## 资料完整性

- 有 `研究计划.md`。
- 有 `资料索引.md`。
- 有阶段日志或等价进度记录。
- 正式结论能追溯到来源。
- 区分了官方文档、源码、论文、二手资料。

## 源码证据

如果 clone 了仓库，必须记录：

- 仓库 URL。
- commit hash。
- clone 策略，例如 shallow clone 或 sparse checkout。
- 阅读过的关键文件路径。
- 哪些部分没有读。

## 论文证据

如果引用论文，必须记录：

- 标题。
- 链接。
- 年份。
- 解决的问题。
- 方法结构。
- 对当前目标的启发和局限。

## 判断质量

检查最终报告是否回答：

- 主流成熟方案是什么？
- 前沿方法是什么？
- 哪些只是研究原型？
- 哪些适合用户当前目标？
- 有无类似现成项目？
- 如果要自研，差异化在哪里？
- 下一步如何落地？

## 内容启发式审计

`check_quality.py` 会先做硬性结构检查，再做内容启发式审计。内容审计默认只输出 warning，不阻断普通 smoke/CI；需要严格评审时再使用 `--strict-content`。

重点看：

- 最终报告是否出现证据等级、来源引用或本地笔记/源码路径。
- 最终报告是否写出 `待验证/不确定/局限/风险/假设`。
- 最终报告是否给出下一步动作、路线图或 MVP。
- 项目笔记是否只停留在 README，还是记录了源码路径、关键文件、入口、测试或扩展点。
- 论文笔记是否包含工程启发、局限、和当前目标的关系。
- PDF 正文抽取是否标记了质量，例如 `正常`、`抽取质量低/需复核`、`疑似扫描版/待 OCR`、`待人工阅读`。

这些规则只能发现浅度风险，不能替代 reviewer 的真实复核。

## Reviewer 复核清单

重要调研建议运行：

```bash
python3 <skill_dir>/scripts/check_quality.py <workspace> --write-report --write-review
```

生成的 `内容复核清单.md` 给 reviewer 角色逐项确认：

- 关键结论是否能追溯到官方文档、论文正文或源码路径。
- 是否有只看 README 或二手博客就下结论的段落。
- 自动发现的搜索入口是否已经二次验证。
- PDF 抽取质量低的论文是否被标为待人工阅读。
- 不确定项和下一步动作是否清楚。

## 风险标注

以下内容必须标注：

- 没有读源码，只读文档。
- 网络无法访问或 clone 失败。
- 只看到旧版本资料。
- 判断来自二手文章。
- 结论依赖用户未确认的假设。

## 外部副作用

默认不得执行：

- `git push`
- 发布到网站/社媒/公众号
- 发送邮件或消息
- 修改远端资源
- 调用会写入外部系统的 API

如果用户另行要求这些动作，把它们当成独立任务处理，不属于本 skill 的默认研究流程。
